Introduction to the Software Engineering Crisis

What does the Ariane 5 rocket, Mars Climate Orbiter, Therac-25 radiation therapy machine, Patriot surface-to-air missile, and Knight Capital all have in common? They have all suffered in some way from software bugs. These software bugs have caused an immense loss of money, and even cost some people their lives. For example, the Ariane 5 rocket flight cost over $370 million USD[1]. In addition on February 25, 1991, a Patriot missile suffered a software error that failed to intercept a Iraqi Scud missile killing 28 United States Army soldiers.[2]

Many people outside the software industry take technology for granted and they do not realize how easy it is to create a bug in the software, and how they can affect common people. A lot of people do not know that a nuclear war almost broke out during the Cold War because of a computer bug. In September 1983. A malfunctioning computer in the Soviet Union's new Oko early-warning system detected five ballistic missile from the United Stated headed towards the Soviet Union. Of course this did not happen! The Soviet officer monitoring the system quickly rationalized that if the United States initiated a nuclear attack they would certainly send more than five missiles[3]. If the Soviet Union followed correct procedure then the world as we know it would be completely different or might have even resulted in the destruction of civilization.

Why is There a Crisis?

The ability to Release Software on Time

  • Schedules designed by someone with less experienced with software engineering will want a product out by a certain time, but this is simply not feasible.
  • Before a project is even started to be coded there is a lot of design and requirement that need to be considered and documented.
    • In the middle of the project a requirement or feature could be added completely changing all previous world.
  • Bad Design can lead to bugs in code causing even more delays.
  • Unrealistic requirements that developers cannot fill.
    • Can else be caused by bad communication between developers and end-users.

Program Code That is Difficult to Maintain and Update

Ever notice how your Window operating system is constantly updating, or your mobile device is always installing upgrade? This is because as software is released developers and managers find bugs within the code or come up with a new feature that would be useful. Most of the time the software developers who wrote the original code are not the ones who are updating the software. So how are these developers supposed to know what the old code is doing, and how to upgrade it?

One method to create readable code is by following coding standards. Using these standards programmers can understand the types of variables, methods, and makes it easy for an one else reading the code. Software is Intangible

When software is delivered to a client they do not receive anything physical. Some client do not understand the actual value it provides to the company, they take it for granted. As an example, people use Microsoft Word all the time and do not realize how much longer and harder it is to type and edit a document. This also causes problems with meeting deadlines.

The developing team responsible software cannot give visible evidence of the progress to the client, unlike civil engineers who can visually show the progress of a bridge being built. This can lead to exaggeration or even lying about deadlines. As with any type of engineering, software engineers must be civil.
